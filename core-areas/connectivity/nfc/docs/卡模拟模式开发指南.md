## 一、基本知识

### 2.1 核心机制 HCE（Host-based Card Emulation）

- **原理**：手机不再依赖物理安全芯片（eSE），而是由**应用处理器（AP）** 直接运行卡逻辑。外部读卡器通过 NFC 射频场供电，唤醒手机中的 `HostApduService`服务来处理指令。
- **通信协议**：**APDU**（Application Protocol Data Unit），而非 NDEF。这是智能卡（ISO 7816）的标准二进制指令集。
- **路由逻辑**：读卡器发送 `SELECT`指令（含 AID）→ Android 系统根据 AID 匹配清单文件 → 唤醒对应的 App 服务。



> 基于主机的卡模拟 (HCE)：依靠手机 AP 处理器运行，通过 HostApduService实现（应用层开发）。
> 安全元件卡模拟 (SE)：依靠手机内置的 eSE 或 SIM 卡安全芯片运行（通常需要系统权限或厂商合作，普通 App 无法直接开发）。



注意，标准 Android HCE 仅支持模拟 ISO 7816-4 协议（即银行卡、交通卡用的那种）。很多传统门禁卡是基于 **Mifare Classic（M1）** 的。对于 M1 卡，HCE 无法直接模拟其底层逻辑（如修改 UID），这通常需要手机厂商在底层 NFC 驱动中开放“门禁模拟”白名单或使用 eSE（嵌入式安全元件）才能实现。

手机厂商通常会在底层做协议转换。即：将读卡器发来的 MIFARE 指令，在手机内部转换成标准的 ISO 14443-4 指令，交给 eSE 处理，再将结果转换回去。这也是为什么很多第三方 App 无法模拟 M1 卡，而手机自带钱包可以的原因。


### 2.2 AID（Application Identifier）

- **作用**：相当于卡的“身份证号”。系统依靠 AID 决定将读卡器的指令路由到哪个 App。
- **格式**：十六进制字符串（如 `F0010203040506`）。
- **分类**：
  - **支付 AID**（Category Payment）：通常受系统严格管控，需设置为默认支付应用。
  - **其他 AID**（Category Other）：用于门禁、会员卡、自定义业务，开发自由度更高。



### 2.3 通信协议数据单元

在EMV支付标准中，终端与芯片卡之间的所有对话都通过一种称为 **APDU（Application Protocol Data Unit，应用协议数据单元）** 的结构化命令来完成。APDU通信采用简单的“一问一答”模式，即终端发送一个**命令APDU（**C-APDU**）**，智能卡返回一个**响应APDU（**R-APDU**）**。

#### 2.3.1 命令 APDU 组成结构

命令 APDU 组成结构如下所示：

```
+-----+-----+----+----+----+------------+----+
| CLA | INS | P1 | P2 | Lc |    Data    | Le |
+-----+-----+----+----+----+------------+----+
|    Mandatory Header |  Conditional Body    |
+-----+-----+----+----+----+------------+----+
```

每一部分的含义为：

| 组成部分   | 描述与作用                                                   | 长度（字节）    |
| ---------- | ------------------------------------------------------------ | --------------- |
| **CLA**    | **指令类别**。指明命令的总体类型，例如是用于支付环境的国际命令（如0x80）还是卡片管理相关的通用命令（如0x00）。 | 1               |
| **INS**    | **指令代码**。指定要执行的具体操作，例如`SELECT`（选择应用）或`GET PROCESSING OPTIONS`（获取处理选项）。 | 1               |
| **P1, P2** | **参数1和参数2**。为指令提供更详细的上下文或选项，例如指定要读取的文件标识。 | 各1             |
| **Lc**     | **命令数据域的长度**。指明紧随其后的**Data**字段包含多少字节的数据。如果命令不携带数据，则该字段可省略。 | 可变（0, 1或3） |
| **Data**   | **命令数据域**。可选字段，包含指令执行所需的具体信息，例如终端传递给卡片的交易数据。 | 由Lc指定        |
| **Le**     | **期望响应长度**。指示终端期望从卡片返回的最大数据长度。如果只关心状态字，该字段可省略。 | 可变（0, 1或3） |



#### 2.3.2 响应 RPDU 组成结构

响应 RPDU 结构如下所示：

```
+--------------+----+----+
|     Data     | SW1| SW2|
+--------------+----+----+
|    Body      | Trailer |
+--------------+----+----+
```

一个标准的响应APDU包含以下几个核心部分：

| 字段 (Code) | 说明 (Description)                       | 长度 (Length) |
| ----------- | ---------------------------------------- | ------------- |
| **Data**    | 响应中接收到的数据字节串（其长度等于Lr） | 可变 (Lr)     |
| **SW1**     | 命令处理状态                             | 1 字节        |
| **SW2**     | 命令处理限定符                           | 1 字节        |

通常，RPDU 中 Data 是 TLV 数据格式，可以使用 http://i.goto327.top:85/EMV/ParseTlv.aspx 来解析。



### 2.4 系统调度逻辑

在 Android 的 HCE（Host Card Emulation）架构中，门禁卡和银行卡在技术实现上并没有本质区别，它们都是通过**应用 ID（AID）**来区分的虚拟卡片。系统的调度逻辑核心是**AID 路由表**和**默认钱包机制**。



#### 2.4.1 技术本质：都是 AID 应用

HCE 允许手机用软件模拟一张智能卡（ISO 7816-4）。无论是银行卡还是门禁卡，底层都是通过 `HostApduService`服务来响应 NFC 读卡器的指令。

| 维度         | 银行卡 (Payment)             | 门禁卡 (Other)                  |
| ------------ | ---------------------------- | ------------------------------- |
| **AID 类别** | `CATEGORY_PAYMENT`           | `CATEGORY_OTHER`                |
| **安全级别** | 高（通常需 SE/Tokenization） | 低（通常仅校验 UID 或简单密钥） |
| **默认行为** | 受“默认钱包”设置严格管控     | 允许多应用共存，优先级较低      |



#### 2.4.2 AID 路由与优先级

当手机靠近读卡器时，读卡器会发送一个 `SELECT AID`指令（包含目标应用 ID）。Android NFC 服务（NfcService）会根据以下逻辑决定将数据包交给哪个应用处理：

1. **构建路由表**：系统启动时，会扫描所有声明了 `<host-apdu-service>`的应用，根据其 `apduservice.xml`中注册的 AID 和 category 构建路由表。
2. **匹配 AID**：NFC 控制器接收到读卡器的 AID 后，查询路由表找到匹配项。
3. **冲突解决**：如果多个应用注册了同一个 AID，系统按以下优先级裁决：
   - **Payment 类别特权**：如果其中一个应用将该 AID 声明为 `PAYMENT`，且该应用是用户设置的**默认钱包**（如 Google Pay、手机厂商钱包），则直接路由给它。
   - **用户选择**：如果都是 `OTHER`类别且 AID 冲突，系统会弹窗让用户选择用哪个应用响应。
   - **前台偏好**：如果某个应用正在前台运行并调用了 `setPreferredService()`，它会获得临时最高优先级。





## 二、Android 核心 API

开发卡模拟 App，你只需要与以下 3 个核心类打交道：

| 类/接口                            | 作用                                                         | 关键方法/属性                                                |
| ---------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **`HostApduService`** *(核心服务)* | **处理 APDU 通信** 你的 App 需要继承此 Service 来接收和响应读卡器指令。 | • `processCommandApdu()`：**核心回调**，处理读卡器发来的 C-APDU。<br>• `sendResponseApdu()`：发送响应数据（R-APDU）。<br>• `onDeactivated()`：会话结束（如手机移开）。 |
| **`CardEmulation`** *(路由管理)*   | **管理 AID 路由与默认应用** 用于查询当前服务的路由状态。     | • `getInstance()`：获取单例。<br>• `isDefaultServiceForCategory()`：检查是否为默认支付应用。<br>• `setPreferredService()`：**前台优先**（尝试让当前 App 优先响应）。 |
| **`NfcAdapter`** *(硬件适配)*      | **检测 NFC 硬件状态**                                        | • `getDefaultAdapter()`：获取适配器。<br>• `isEnabled()`：检查 NFC 是否开启。 |

同时，`PackageManager.hasSystemFeature()` 可以检测设备硬件是否支持某项功能。





### 2.1 权限与特性声明（AndroidManifest.xml）

```xml
<!-- 声明需要 NFC 和 HCE 功能 -->
<uses-feature android:name="android.hardware.nfc.hce" android:required="true" />
<uses-permission android:name="android.permission.NFC" />

<!-- 注册 HCE 服务（注意 exported 和 permission） -->
<service
    android:name=".MyHostApduService"
    android:exported="true"
    android:permission="android.permission.BIND_NFC_SERVICE">
    <intent-filter>
        <action android:name="android.nfc.cardemulation.action.HOST_APDU_SERVICE" />
    </intent-filter>
    <!-- 指向 AID 路由配置文件 -->
    <meta-data
        android:name="android.nfc.cardemulation.host_apdu_service"
        android:resource="@xml/apdu_service_config" />
</service>
```



### 2.2  配置 AID 路由（res/xml/apdu_service_config.xml）

```xml
<host-apdu-service
    xmlns:android="http://schemas.android.com/apk/res/android"
    android:description="@string/service_name">
    
    <!-- 支付类 AID（通常需要系统级权限） -->
    <aid-group android:category="payment">
        <aid-filter android:name="A0000000031010" /> <!-- 示例 AID -->
    </aid-group>
    
    <!-- 你的自定义业务 AID（推荐使用） -->
    <aid-group android:category="other">
        <aid-filter android:name="F00102030405" />
    </aid-group>
</host-apdu-service>
```



### 2.3 实现 APDU 处理逻辑

```kotlin
class MyHostApduService : HostApduService() {
    override fun processCommandApdu(commandApdu: ByteArray?, extras: Bundle?): ByteArray? {
        // 1. 解析 commandApdu（CLA, INS, P1, P2...）
        // 2. 根据指令执行业务逻辑
        // 3. 返回响应数据 + SW1 SW2（如 0x90 0x00 表示成功）
        return byteArrayOf(0x90, 0x00) 
    }
    
    override fun onDeactivated(reason: Int) {
        // 读卡器断开连接，清理会话
    }
}
```



### 2.4 运行时检查与路由控制

```kotlin
// 检查设备是否支持 HCE
val pm = context.packageManager
val hasHCE = pm.hasSystemFeature(PackageManager.FEATURE_NFC_HOST_CARD_EMULATION)

// 获取 CardEmulation 实例，管理路由
val cardEmulation = CardEmulation.getInstance(NfcAdapter.getDefaultAdapter(context))

// 查询当前服务是否为默认支付应用
val isDefault = cardEmulation.isDefaultServiceForCategory(
    ComponentName(context, MyHostApduService::class.java),
    CardEmulation.CATEGORY_PAYMENT
)
```





## 三、最佳实践

- **AID 冲突**：确保你的 AID 不与系统预置的支付 AID 冲突，否则可能无法被选中。
- **前台优先策略**：在 Activity 中调用 `setPreferredService()`可临时提升优先级，但退出时必须调用 `unsetPreferredService()`。
- **性能与超时**：`processCommandApdu()`执行时间不宜过长（通常需在 500ms 内响应），否则读卡器会报超时错误。
