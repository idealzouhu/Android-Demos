## 一、检查性能

这是**开发者视角**的操作。当你在本地运行 App，发现动画卡顿、内存暴涨时，你需要“深入检查”内部发生了什么。

- **操作**：你连接真机，打开 Android Studio Profiler 或 Perfetto，录制一段操作，然后像法医解剖一样分析每一帧的渲染时间、每一个方法的 CPU 耗时。
- **目的**：找到具体的代码行、不合理的算法、未释放的引用。
- **工具**：**Perfetto**（系统级追踪）、**Android Profiler**（应用级剖析）、**LeakCanary**（自动内存泄漏检测）。



### 关键工具

查看 https://developer.android.com/topic/performance/inspecting-overview#app-startup-performance

### 1.1 性能分析技术

| 维度         | [Tracing（追踪）](https://perfetto.dev/docs/tracing-101#introduction-to-tracing) | [Profiling（剖析）](https://perfetto.dev/docs/tracing-101#introduction-to-profiling) | Metrics（指标）          |
| ------------ | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------ |
| **本质**     | 高细粒度的事件时间线记录                                     | 基于采样的资源使用统计                                       | 数值化的系统状态快照     |
| **数据形式** | Trace（轨迹文件），包含事件上下文                            | Profile（剖析文件），包含调用栈                              | 时间序列数值（如 CPU %） |
| **优势**     | **全链路还原**，无需复现 Bug                                 | **低开销**，定位热点函数                                     | **极低成本**，长期监控   |
| **劣势**     | 数据量大，开销相对高                                         | 缺乏“为什么”执行的上下文                                     | 无法进行根因分析         |
| **适用场景** | 根因分析、性能问题深挖                                       | CPU/内存热点分析                                             | 生产环境监控、告警       |

## 二、监控性能

这是**产品/质量视角**的操作。当 App 发布到线上，你需要持续“监控”它在成千上万台不同设备上的表现。

- **操作**：集成监控 SDK（如 Firebase Performance Monitoring），或者运行自动化基准测试（Benchmark）。
- **目的**：发现趋势性恶化（如“最近一次发布后，冷启动时间增加了 200ms”），收集真实用户的 ANR 和 Crash 数据。
- **工具**：**Android Vitals**（Google Play 后台数据）、**Firebase**、**Macrobenchmark**（基准测试库）。
